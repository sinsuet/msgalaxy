## 5. 为什么是 `PARTIAL_SUCCESS`，不是 `SUCCESS`

这是本次结果最容易误解的点。

### 5.1 可行判据（求解层）

`diagnose_solver_outcome` 判据核心是：

- 若 `best_cv <= 1e-9`（或存在可行目标曲线）→ 可判为 `feasible` / `feasible_but_stalled`。

本次第 8 次 attempt 满足 `best_cv=0.0`，所以求解诊断是 `feasible`。

### 5.2 运行状态判据（流程层）

`workflow/maas_pipeline_service.py` 里最终 `SUCCESS` 还要求：

- `diagnosis_status in {"feasible","feasible_but_stalled"}`
- 且 `len(final_violations)==0`

本次 `evolution_trace.csv` 最终记录：

- `num_violations = 1`
- `penalty_violation = 100`
- `penalty_clearance = 50`
- `penalty_score = 150`

所以最终状态是 `PARTIAL_SUCCESS`。

### 5.3 根因（本次数据）

对 `g_clearance`：

- 初始硬阈值: `min_clearance >= 6.0mm`
- 最终候选值: `min_clearance = 5.0mm`
- 在原阈值下: `g_clearance = 6.0 - 5.0 = +1.0`（违规）

第 8 次 attempt 实际使用的是放松后阈值（见 `maas_diagnostics.jsonl`）：

- `g_clearance target = 4.935`
- 对应 `g_clearance = 4.935 - 5.0 = -0.065`（可行）

因此出现“求解可行，但运行状态部分成功”的并存结果。

### 6.3 `attempts.csv`（Attempt 级）

字段分组解释：

- 运行上下文: `run_id/timestamp/iteration/attempt`
- 分支信息: `branch_action/branch_source/search_space_mode`
- 求解诊断: `diagnosis_status/diagnosis_reason/solver_message`
- 代价与得分: `solver_cost/score/best_cv/aocc_cv/aocc_objective`
- 违规聚合: `dominant_violation/violation_keys/violation_total`
- 候选指标快照: `metric_*`（cg/temp/clearance/struct/power）

### `best_cv`（最关键）

- 来自当前 attempt 的 `best_cv_curve` 最小值，表示最优个体总约束违背程度。
- `0` 代表可行，`>0` 代表仍有违规。

### `aocc_cv` 与 `aocc_objective`

由 `calculate_aocc` 计算（归一化曲线面积）：

- 值域近似 `[0,1]`
- 越高表示“收敛越快”
- `aocc_cv`: 基于约束违背曲线
- `aocc_objective`: 基于可行目标曲线（无可行点时常为 0）

### `score`（MCTS 回传分）

由多项组成（`workflow/orchestrator.py`）：

- `status_base`（例如 feasible=1500, no_feasible=320）
- `aocc_cv_bonus = aocc_cv * 140`
- `aocc_obj_bonus = aocc_objective * 90`
- `best_cv_penalty = -best_cv * 65`
- `clearance_bonus = min(min_clearance,30)*3`
- `cg_penalty = -cg_offset*1.8`
- `collision_penalty = -num_collisions*320`
- `solver_cost_penalty = -solver_cost*0.15`
- `first_feasible_bonus = max(0, 8-attempt)*95`（若可行）

这就是第 8 次分数 1717 显著高于前 7 次（约 373~404）的直接原因。

### 6.4 `generations.csv`（代际收敛）

关键字段：

- `generation`: 代号
- `population_size`: 种群数（本次固定 48）
- `feasible_count / feasible_ratio`: 代内可行个体数量/比例
- `best_cv / mean_cv`: 代内最优/均值约束违背
- `best_feasible_sum_f`: 代内可行个体目标和（可行时才有意义）

如何读：

- 先看 `best_cv` 是否降到 0。
- 再看 `feasible_ratio` 是否从 0 上升并稳定在高位。

## 7. 本次运行“关键数字”解释

### 7.1 初始状态 vs 最终候选

初始（`iter_01_context.json`）：

- `cg_offset=29.9485`（超 16mm）
- `min_clearance=5.0`（低于 6mm）
- `max_temp=35.6012`
- 初始违规 2 条（clearance + cg）

最终候选（attempt8）：

- `cg_offset=1.8689`（显著改善）
- `min_clearance=5.0`（未改善）
- `max_temp=35.7047`（基本持平）
- `safety_factor=3.8643`、`first_modal_freq=87.9426`（结构代理明显改善）

结论：这次成功点是“把综合几何/结构/电源推到可行区附近并通过放松阈值过线”，而不是“提升间隙到 6mm”。

### 7.2 为什么 `g_clearance` 一直主导

从 `attempts.csv`：

- 前 7 次 `dominant_violation = g_clearance`
- `metric_min_clearance` 始终约 5.0
- `best_cv` 从 1.0 降到 0.64，再到 0.0（仅在阈值进一步放松后）

这说明几何布局在该初始化/搜索轨迹下难以自然突破 6mm 间隙线，需策略性放松与再探索。

### 7.3 `mass_trace.csv` 与 `tables/attempts.csv` 计数差异

- `mass_trace.csv` 有 9 行（attempt=8 出现两次）
- `tables/attempts.csv` 有 8 行（严格 attempt 事件）

原因：`mass_trace.csv` 会附加一个“最佳 attempt 标记行”（`is_best_attempt=True`），用于可视化高亮。
